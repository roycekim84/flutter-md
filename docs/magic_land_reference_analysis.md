# 마법의대륙 롤모델 분석과 적용안

## 1. 전제

이 문서는 옛 한국 텍스트 MUD 게임인 마법의대륙을 직접 복제하기 위한 문서가 아니다.

목표는 사용자가 기억하는 마법의대륙의 감각을 분석하고, Flutter web으로 만드는 새 텍스트 MUD에 적용할 설계 원칙을 정리하는 것이다. 공개 검색에서는 마법의대륙 자체에 대한 신뢰도 높은 1차 자료를 충분히 확인하지 못했으므로, 특정 시스템을 사실처럼 단정하지 않는다. 이후 사용자의 기억, 스크린샷, 명령어 목록, 플레이 경험을 받아 이 문서를 계속 보강한다.

## 2. 참고 가능한 MUD 장르 특성

전통적인 MUD는 텍스트 기반의 실시간 가상 세계이며, 플레이어는 방 설명, 사물, NPC, 다른 플레이어 정보를 읽고 자연어에 가까운 명령어를 입력해 행동한다. 일반적인 이동은 `north`, `n` 같은 방향 명령으로 처리되고, 사물 획득이나 전투도 `take apple`, `attack dragon` 같은 동사 중심 명령으로 이뤄진다.

마법의대륙을 롤모델로 삼을 때 핵심은 다음 네 가지다.

- 방 기반 세계
- 명령어 중심 조작
- 전투와 성장의 반복 루프
- 텍스트로 형성되는 장소감과 커뮤니티 감각

## 3. 마법의대륙에서 가져오고 싶은 감각

사용자가 기억하는 롤모델의 핵심을 아래 질문으로 구체화한다.

| Area | 확인할 질문 | 우리 게임에 필요한 이유 |
| --- | --- | --- |
| 첫 경험 | 처음 접속했을 때 어떤 문구, 장소, 감정이 기억나는가? | 첫 화면과 시작 지역 설계 |
| 명령어 | 자주 썼던 명령어가 무엇인가? | 명령어 별칭과 UX 설계 |
| 이동 | 방향 이동이 중심이었는가, 지역명 이동이 있었는가? | 월드 그래프 설계 |
| 사냥 | 사냥은 자동 반복이었는가, 매번 명령어 입력이었는가? | 전투 피로도 조절 |
| 성장 | 레벨, 직업, 스킬, 능력치 중 무엇이 중요했는가? | 성장 시스템 우선순위 |
| 아이템 | 장비 파밍이 중요했는가, 소비 아이템이 중요했는가? | 보상 설계 |
| 커뮤니티 | 파티, 문파, 채팅, 장터 중 무엇이 기억나는가? | 싱글플레이에서 대체할 요소 |
| 위험 | 사망 패널티, 길 잃음, 고레벨 지역 공포가 있었는가? | 긴장감 설계 |
| 운영 | 운영자 이벤트나 공지 감성이 있었는가? | 라이브 로그와 이벤트 연출 |

## 4. 적용 방향

### 4.1 명령어는 짧고 관대하게 처리한다

옛 MUD의 재미는 명령어를 외우는 데서도 오지만, 웹 게임의 첫 진입 장벽은 낮아야 한다.

적용안:

- 한글 명령어를 1순위로 지원
- 영문 MUD식 축약 명령어도 지원
- 띄어쓰기 실수를 허용
- 방향 명령은 가장 짧게 입력 가능하게 한다
- 잘못된 명령어에는 가능한 후보를 제안한다

예시:

| 입력 | 처리 |
| --- | --- |
| `북` | 북쪽 이동 |
| `ㅂ` | 북쪽 이동 |
| `n` | 북쪽 이동 |
| `이동 북쪽` | 북쪽 이동 |
| `고블린 공격` | 공격 고블린으로 보정 |
| `물약먹기` | 사용 물약으로 보정 |

### 4.2 텍스트 로그를 게임의 중심 화면으로 둔다

마법의대륙 같은 텍스트 MUD의 핵심은 그래픽이 아니라 로그의 누적이다. Flutter UI는 로그를 보조하는 역할이어야 한다.

적용안:

- 로그 패널이 화면의 60% 이상을 차지한다.
- 장소 설명, 전투 결과, 시스템 메시지를 색으로 구분한다.
- 최신 로그만 강조하고 오래된 로그는 살짝 흐리게 표시한다.
- 중요한 단어는 클릭 가능한 토큰으로 만들어 명령어 입력을 보조한다.

로그 타입:

| Type | Color Intent | Example |
| --- | --- | --- |
| location | 장소 설명 | 루미르 광장에 도착했다. |
| combat | 전투 | 당신은 이끼쥐에게 4의 피해를 입혔다. |
| reward | 보상 | 7 경험치와 3 골드를 얻었다. |
| warning | 위험 | 폐광 안쪽에서 낮은 울음소리가 들린다. |
| system | 시스템 | 저장이 완료되었다. |
| chatlike | NPC/이벤트 | 경비대장 에란: 폐광 쪽으로는 가지 않는 게 좋소. |

### 4.3 지역은 좁게 시작하되 밀도 있게 만든다

옛 MUD의 세계는 거대한 방 개수보다 “여기서 뭘 할 수 있는지”가 중요하다. MVP에서는 방 수를 늘리기보다 각 방에 반복 방문 이유를 둔다.

적용안:

- 시작 지역은 12개 이하의 방으로 제한한다.
- 각 방에는 최소 하나의 기능을 둔다.
- 같은 방 설명도 상태에 따라 바뀌게 한다.
- 퀘스트 진행, 시간대, 처치 여부에 따라 방의 문장이 달라진다.

예시:

| Room | First Visit | Later Visit |
| --- | --- | --- |
| 루미르 광장 | 공지와 학자 소개 | 공지, 소문, 귀환 지점 |
| 헤스틴 여관 | 회복 장소 소개 | 사망 후 부활, 소문 획득 |
| 오르페 폐광 입구 | 위험 경고 | 퀘스트 체크포인트 |
| 별빛 수직갱 | 보스 암시 | 반복 사냥과 아이템 파밍 |

### 4.4 전투는 반복 가능하지만 완전히 자동화하지 않는다

텍스트 MUD식 사냥은 반복이 매력일 수 있지만, 웹 싱글플레이에서는 지루해질 수 있다. 따라서 초반은 간단한 턴제 전투로 만들고, 이후 명령어 매크로와 자동 사냥 느낌을 제한적으로 제공한다.

MVP 적용안:

- `공격 [대상]`으로 1턴 진행
- 같은 대상과 전투 중이면 `공격`만 입력해도 계속 공격
- `도망` 명령 추가
- 체력이 낮으면 경고 로그 출력
- 사냥 반복을 위해 `다시` 명령을 제공

확장 적용안:

- `연속공격 3`처럼 제한된 반복 명령 지원
- 스태미나 또는 집중도 같은 자원으로 무한 반복 방지
- 특정 지역에서만 자동 반복 사냥 허용

### 4.5 커뮤니티 감각은 싱글플레이에서도 흉내 낸다

MVP는 싱글플레이지만, 마법의대륙의 롤모델 감각에는 “나 혼자만 있는 세계가 아닌 느낌”이 중요할 수 있다.

적용안:

- NPC 대사를 채팅 로그처럼 출력한다.
- 마을 광장에 가짜 공지와 소문 로그를 표시한다.
- 유령 플레이어 같은 흔적을 만든다.
- 랭킹보다는 “오늘의 소문”, “누군가 남긴 표식” 같은 비동기 감성을 쓴다.

예시:

```text
[소문] 어젯밤 오르페 폐광 입구에서 낡은 투구를 본 사람이 있다.
[공지] 루미르 수비대는 서쪽 폐광으로 향하는 여행자에게 주의를 당부합니다.
[흔적] 누군가 벽에 "동쪽은 아직 이르다"라고 긁어 두었다.
```

### 4.6 사망과 위험은 약하게라도 남긴다

옛 MUD의 긴장감은 길을 잘못 들거나 사망했을 때 손해가 있다는 데서 온다. 단, MVP에서는 너무 큰 손실을 주지 않는다.

적용안:

- 사망 시 여관에서 부활
- 현재 골드의 일부를 잃는다.
- 경험치는 잃지 않는다.
- 위험 지역 진입 전 경고 로그를 출력한다.
- 첫 사망 때는 튜토리얼처럼 처리한다.

```text
눈앞이 어두워졌다.
낡은 여관의 침대에서 깨어났다.
주머니가 조금 가벼워진 것 같다.
```

## 5. 기존 기획에 반영할 변경점

현재 `text_mud_game_plan.md`에 다음을 반영한다.

- 게임의 롤모델을 마법의대륙 계열 한국 텍스트 MUD로 명시한다.
- 명령어에 `도망`, `다시`, `소문`, `공지`를 추가한다.
- 로그 타입을 데이터 모델에 추가한다.
- 방 설명을 정적 문자열이 아니라 상태 기반 설명으로 확장할 여지를 둔다.
- MVP에도 커뮤니티 감각을 흉내 내는 시스템 로그를 넣는다.
- 전투 중 대상 기억 기능을 추가한다.

## 6. Flutter 구현 관점

### 6.1 CommandParser

명령어 파서는 정확한 문장 분석보다 별칭 테이블과 보정 규칙이 중요하다.

```dart
class ParsedCommand {
  final CommandType type;
  final String? target;
  final Map<String, String> args;
}
```

필요한 처리:

- 한글 초성 또는 약칭 매핑
- 영문 축약어 매핑
- 붙여쓴 명령어 분리
- 대상명 부분 일치
- 실패 시 추천 명령어 반환

### 6.2 GameLog

로그는 단순 문자열이 아니라 타입과 메타데이터를 갖는다.

```dart
enum GameLogType {
  location,
  combat,
  reward,
  warning,
  system,
  npc,
  rumor,
}

class GameLogEntry {
  final GameLogType type;
  final String message;
  final DateTime createdAt;
  final List<String> tags;
}
```

### 6.3 Room Description

방 설명은 함수 또는 조건 목록으로 확장할 수 있게 설계한다.

```dart
class RoomDescriptionRule {
  final String id;
  final String text;
  final bool Function(GameState state) condition;
}
```

MVP에서는 Dart 상수로 시작하고, 나중에 JSON 또는 YAML로 분리한다.

## 7. 우선순위

1. 마법의대륙식 첫 화면 감성 만들기
2. 짧은 한글 명령어와 영문 축약 명령어 지원
3. 로그 중심 UI 완성
4. 방 기반 이동과 상태 기반 설명
5. 기본 전투와 사망 처리
6. NPC/소문/공지로 커뮤니티 감각 추가
7. 반복 사냥 명령과 아이템 파밍

## 8. 사용자에게 확인할 내용

다음 정보가 있으면 롤모델 분석 정확도를 크게 높일 수 있다.

- 기억나는 마법의대륙 시작 문구
- 자주 쓰던 명령어
- 직업 또는 종족이 있었는지
- 전투가 어떤 방식이었는지
- 사망하면 무엇을 잃었는지
- 채팅, 파티, 문파, 거래 중 무엇이 중요했는지
- 가장 기억나는 지역명, 몬스터명, 아이템명
- 플레이 화면 색감이나 터미널 스타일
